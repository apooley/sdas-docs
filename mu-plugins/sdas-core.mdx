## Overview

SDAS Core is a must-use (MU) plugin that centralizes shared infrastructure for all SDAS feature plugins. It provides common services for error handling, observability, OpenAI access, security, storage, and admin UX so individual plugins stay focused on business logic instead of plumbing.

Use this page when you need to understand which cross-cutting concerns SDAS Core owns, how it loads into WordPress, and which Core services your plugin should call instead of re-implementing its own helpers.

<Callout kind="info">

WordPress loads MU-plugins on every request, before regular plugins. SDAS Core uses a small loader file at `wp-content/mu-plugins/sdas-core-loader.php` that checks `ABSPATH`, then attempts to require `wp-content/mu-plugins/sdas-core/sdas-core.php`.

</Callout>

## Loader behavior and plugin layout

SDAS Core consists of:

- A loader file at `wp-content/mu-plugins/sdas-core-loader.php`
- The main plugin directory at `wp-content/mu-plugins/sdas-core/`, including `sdas-core.php` and supporting modules

On each request:

1. `sdas-core-loader.php` verifies that WordPress is bootstrapped by checking the `ABSPATH` constant.
2. The loader computes the path to `wp-content/mu-plugins/sdas-core/sdas-core.php`.
3. If that file exists, the loader requires it, bootstrapping all SDAS Core services.
4. If the file or directory is missing, the loader registers an `admin_notices` callback that displays an error banner in the WordPress admin.

## Core capabilities by area

Use the tabs below to understand SDAS Core capabilities grouped by concern: shared services, admin UX, and operational tooling.

<Tabs>

<Tab title="Services" icon="code">

SDAS Core exposes a set of runtime services that SDAS feature plugins should use for cross-cutting concerns.

### Error handling and logging

Core owns the global error handling stack, including a centralized error handler, log manager, and shutdown handler for fatal errors.

Route all structured logging through the `sdas_log()` function:

```php
<?php
// Example: logging from an SDAS feature plugin
sdas_log('info', 'Started processing daily sync', [
    'source'   => 'crm_import',
    'trace_id' => 'trc_c94d7cfa9e3f4ab0',
]);

try {
    // Feature logic...
} catch (Throwable $e) {
    sdas_log('error', 'Sync failed', [
        'message' => $e->getMessage(),
        'code'    => $e->getCode(),
    ]);
}
```

### OpenAI client

SDAS Core provides a shared OpenAI API client via `SDAS_OpenAI_Client`. This ensures consistent API key management and error handling across the suite.

```php
<?php
$client = new SDAS_OpenAI_Client();
$response = $client->chat([
    'model' => 'gpt-4.1-mini',
    'messages' => [
        ['role' => 'user', 'content' => 'Generate a vehicle description...']
    ]
]);
```

### Secure token storage

Core manages secure storage for credentials and API tokens via `SDAS_Secure_Token_Storage`. Plugins should use this for sensitive data like CRM API keys.

### Input validation

The `SDAS_Input_Validator` class provides a consistent way to normalize and validate request data across all SDAS plugins.

</Tab>

<Tab title="Admin UX" icon="layout">

SDAS Core provides the unified admin interface for the entire plugin suite.

### Centralized Menu

The `SDAS_AdminMenuManager` consolidates all SDAS-related settings and tools under a single top-level **SDAS** menu. Individual plugins register their submenus via the `sdas_register_admin_menu` action.

### Shared Settings

The central **SDAS Settings** page (`sdas-settings`) includes tabs for:
- **System**: Global performance and logging settings.
- **Dealership**: Shared dealership identity data (Name, Address, Logo).
- **Schema**: Centralized structured data management and reporting.
- **Calculator**: Global finance calculator presets.

### Quick Actions

A floating **Quick Actions** menu is available across the WordPress admin, providing one-click access to common tasks like adding vehicles, running imports, or clearing cache.

</Tab>

<Tab title="Ops & Reliability" icon="activity">

SDAS Core includes tools for monitoring and maintaining site health.

### Health Monitor

The `SDAS_Health_Monitor` runs daily checks on database connectivity, queue state, and external API reachability. Results are surfaced in the **SDAS -> Health** page.

### Schema Manager

The `SDAS_Schema_Manager` coordinates structured data output across the site. It provides a **Schema Console** in the settings page that reports on schema ownership and status for all site surfaces (Front Page, Vehicles, Jobs, etc.).

### Database Indexes

The `SDAS_Database_Indexes` class manages custom database indexes for SDAS tables to ensure high-performance queries even with large inventories.

</Tab>

</Tabs>

## Schema emission system

SDAS Core implements a centralized schema emission system that ensures high-quality structured data for dealership-specific content while maintaining compatibility with third-party SEO plugins.

### Implementation in MU-Plugin

The schema system is built directly into the MU-plugin to ensure it loads early and consistently across all requests. It consists of two primary components:

1.  **`SDAS_Schema_Manager`**: Responsible for generating and emitting the core site graph (Organization and WebSite nodes) on the front page. It uses data from the **Dealership Settings** (Legal Name, Address, Phone, Logo) to build a valid `AutoDealer` node.
2.  **`SDAS_SEO_Compatibility`**: An orchestration layer that detects the active SEO environment and applies the appropriate policy for indexation, canonicals, and schema suppression.

### SEOPress integration and overrides

SDAS Core is designed to co-exist with **SEOPress** in a layered ownership model. The `SDAS_SEO_Compatibility` class detects the active SEO adapter at runtime via `detect_adapter()` and applies the appropriate policy for schema emission, indexation, and canonicals.

<Callout kind="warning">

**Important**: When SEOPress is detected as the active adapter, `SDAS_Schema_Manager` defers front page schema output entirely to SEOPress. The `should_output_schema()` method returns `false` when `SDAS_SEO_Compatibility::detect_adapter() === 'seopress'`, meaning SDAS will **not** emit its own `AutoDealer`/`WebSite` graph on the front page. In this configuration, you must ensure SEOPress is correctly configured to emit the dealership's Organization schema.

</Callout>

#### Where SDAS takes ownership (SEOPress suppressed)

On SDAS-owned surfaces other than the front page, the `SDAS_SEO_Compatibility` layer actively suppresses SEOPress's automatic schema generation to prevent duplicate or conflicting JSON-LD blocks. This is achieved by hooking into SEOPress filters (e.g., `seopress_get_json_data_organization`, `seopress_schemas_auto_job_json`) and returning an empty array or string.

SDAS takes ownership on the following surfaces:
- **Vehicle Single (VDP)**: Emits detailed `Car` or `Vehicle` schema.
- **Vehicle Archive**: Emits `ItemList` schema for indexable inventory views.
- **Landing Pages**: Emits structured content schema.
- **Job Postings**: Emits `JobPosting` schema.
- **FAQ Pages**: Emits `FAQPage` schema.

#### Where SEOPress retains full control

- **Front Page (when SEOPress is active)**: As noted above, SDAS defers entirely to SEOPress for front page schema when SEOPress is detected. SEOPress is responsible for emitting the `Organization`/`AutoDealer` graph.
- **Standard Pages/Posts**: SEOPress retains full control over schema for standard WordPress pages and blog posts.
- **Titles and Meta**: SDAS does not override SEOPress's title or meta description logic, except for specific inventory archive requests where it may enforce a `noindex,follow` policy for non-canonical filtered views.
- **Sitemaps**: SDAS relies on SEOPress for XML sitemap generation, though it provides the necessary data to ensure SDAS custom post types are included correctly.

#### Sites without SEOPress

When no supported SEO adapter is detected, `SDAS_Schema_Manager` takes full ownership of the front page schema and emits the `AutoDealer` and `WebSite` graph directly via `wp_head`.

### Schema Console

Developers can monitor the schema state via the **Schema** tab in SDAS Settings. This console provides:
- **Surface Ownership**: A report on which plugin owns the schema for each site surface.
- **Graph Preview**: A visual representation of the generated JSON-LD graph.
- **Warnings**: Alerts for missing required fields (e.g., missing dealership phone or address) that would invalidate the schema.
