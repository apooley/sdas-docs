## Plugin responsibilities and bootstrap

SDAS Vehicle Filters provides AJAX-powered filtering for vehicle inventory and passes URL and click-id context into the filter requests so you can track how users reach specific inventory views.

The plugin bootstrap file is:

```bash
wp-content/plugins/sdas-vehicle-filters/sdas-vehicle-filters.php
```

This file registers the plugin, loads dependencies, and wires the public AJAX endpoints and hooks defined in `includes/AjaxHandler.php`.

<Callout kind="info">

SDAS Vehicle Filters does not own the inventory data. It filters and re-renders existing inventory provided by the SDAS Inventory plugin and your front-end implementation.

</Callout>

For front-end integration details, see the [Vehicle Filters (front end)](/plugins/sdas-vehicle-filters-front-end) guide.

## Public AJAX actions

The plugin exposes two AJAX actions that are registered for both authenticated and unauthenticated users.

<Callout kind="info">

Both actions are available as `wp_ajax_{action}` and `wp_ajax_nopriv_{action}` so they work for logged-in users and public visitors.

</Callout>

### `sdas_vf_filter_vehicles`

Filters the vehicle inventory based on the submitted criteria and returns an updated inventory fragment together with context derived from the current URL and click identifiers.

Use this action when the user changes filter controls such as make, model, price, or body style.

<Callout kind="tip">

Pass any necessary tracking identifiers (for example, click IDs) in the AJAX request so they can be attached to the `sdas_filters_applied` action payload.

</Callout>

### `sdas_vf_get_models`

Returns the list of models for a given make so the front end can dynamically populate dependent filter controls.

Use this action when the user selects or changes the vehicle make and you need the corresponding models without reloading the page.

## Developer Hooks

### Action: `sdas_filters_applied`

When a filter request is processed in `includes/AjaxHandler.php`, the plugin fires an action so other plugins or mu-plugins can react to the applied filters and resulting inventory.

The hook signature is:

```php
do_action(
    'sdas_filters_applied',
    [
        'filters'         => ...,
        'result_count'    => ...,
        'click_ids'       => ...,
        'target_selector' => ...,
    ]
);
```

The payload is a single associative array with these keys:

<ParamField body="filters" param-type="array" required="true" show-location="false">

Normalized filter values applied to the inventory query. The exact structure matches what the filters front end and inventory layer agree on.

</ParamField>

<ParamField body="result_count" param-type="int" required="true" show-location="false">

Total number of vehicles returned by the filtered query.

</ParamField>

<ParamField body="click_ids" param-type="array" show-location="false">

One or more tracking identifiers associated with the current request, typically derived from URL parameters or data attributes on the clicked element.

</ParamField>

<ParamField body="target_selector" param-type="string" show-location="false">

CSS selector for the DOM container that the front end updates with the filtered inventory markup.

</ParamField>

### Filter: `sdas_filter_query_args`

Allows developers to modify the `WP_Query` arguments before the vehicle search is executed. This is useful for adding custom meta queries or adjusting the default sorting.

```php
add_filter('sdas_filter_query_args', function($query_args, $filters) {
    // Example: Exclude vehicles marked as 'private'
    $query_args['meta_query'][] = [
        'key'     => 'is_private',
        'value'   => 'no',
        'compare' => '='
    ];
    return $query_args;
}, 10, 2);
```

### Filter: `sdas_vehicle_filters_normalized_context`

Allows developers to modify the request context, including active filters and sorting, before it is used for rendering or querying.

```php
add_filter('sdas_vehicle_filters_normalized_context', function($context) {
    // Example: Force a specific sort order for a specific page
    if (is_page('special-offers')) {
        $context['active_sort'] = 'price_asc';
    }
    return $context;
});
```

## Performance & Caching

SDAS Vehicle Filters uses a robust caching system based on WordPress Transients to ensure fast response times for AJAX requests.

- **Query Results**: Cached for 15 minutes by default.
- **Filter Ranges & Counts**: Cached for 1 hour by default.
- **Cache Invalidation**: The entire filter cache is automatically cleared whenever a vehicle post is created, updated, or deleted. This ensures that visitors always see accurate inventory data.

Developers can manually clear the cache using the `sdas_vf_cleanup_cache` hourly cron job or by calling `Plugin::getInstance()->cache_manager->clearAll()`.

## Health Monitoring

The plugin integrates with the SDAS Core health check system. It monitors:
- **Dependency Status**: Ensures SDAS Inventory is active.
- **AJAX Endpoints**: Verifies that all required AJAX actions are correctly registered.
- **Widget Registration**: Tracks the number of active filter widgets.

Health status can be viewed in the WordPress admin under **SDAS -> Health**.
