## What Tracking Hub does on your site

SDAS Tracking Hub sends key site events (page views, inventory activity, lead submissions) from your WordPress site to analytics and ads platforms in the background.

You do not manage tags, pixels, or JavaScript snippets for Google Analytics, Google Ads, or Meta on each page. Instead, Tracking Hub runs server-side and uses:

- A small front-end script from the plugin to notice important actions.
- A WordPress REST endpoint on your site to pass those actions to the server.
- Server-side logic to send those events to connected platforms.

From an admin perspective, your main responsibilities are:

- Deciding **which pages and actions matter**.
- Making sure **consent and privacy rules** are followed.
- Keeping **URLs and click IDs stable** for accurate attribution.
- Running **basic QA** in staging and production.

<Callout kind="info">

SDAS Tracking Hub centralizes tracking for analytics and ads platforms, but privacy still starts in the browser. As a site admin, treat tracking as optional per visitor, avoid exposing sensitive data in URLs, and align your setup with your legal and privacy policies.

</Callout>

## What visitors see (and do not see)

Visitors never see "Tracking Hub" by name. They notice:

- **Fast pages without multiple trackers**  
  Because Tracking Hub runs server-side, you avoid loading several third-party tracking scripts on every page.

- **One consistent consent banner or settings panel**  
  Whatever tool you use for consent (core SDAS banner, cookie notice plugin, custom UI) is the *only* visible signal about tracking.

- **Normal inventory and lead flows**  
  Inventory browsing, filters, and lead or contact forms behave as usual. Tracking attaches to events that already happen in those flows.

If visitors change their consent, Tracking Hub adjusts silently in the background. The site should feel the same with tracking on or off.

## What you control as a WordPress admin

As a WordPress administrator or site builder, you influence how well Tracking Hub works through a few key decisions.

### Which pages and actions matter

Work with your marketing or analytics owner to decide:

- **Important pages**  
  - Inventory list pages (search results, category pages).
  - Vehicle detail pages (VDP).
  - Lead and contact pages.

- **Important actions**  
  - Viewing a filtered inventory list.
  - Opening a vehicle detail page.
  - Submitting a lead form or contact request.

You do not have to name events or write code yourself, but you should:

- Keep templates for these pages consistent.
- Avoid changing slugs and URL structures without coordinating.
- Confirm that each important action is clearly tied to a page or form that Tracking Hub can see.

### Consent and privacy expectations

Consent management controls whether the server is allowed to send events to Google Analytics, Google Ads, and Meta at all.

On your site:

- **Assume tracking is conditional**  
  Every page and form should work perfectly even if a visitor says no to tracking. Tracking should never block a form from submitting or a page from loading.

- **Use a single consent source of truth**  
  SDAS Core and Tracking Hub expose helpers so the system can check whether tracking is allowed. Avoid multiple, disconnected consent toggles in different plugins.

- **Avoid PII in URLs or visible identifiers**  
  Do not include full names, emails, or phone numbers in URL paths or query strings. Tracking Hub handles any necessary user data server-side; your front end should only expose generic or inventory-related context.

- **Make consent easy to change**  
  Ensure the banner or settings panel is easy to find and use. When visitors update their choice, the site should respect that change from that point forward.

<Callout kind="info">

If your legal or privacy team defines specific rules (for example, "no marketing tracking until marketing consent is granted"), share those requirements with whoever configures SDAS Tracking Hub so the WordPress settings and consent logic match your policy.

</Callout>

### How the REST tracking endpoint is used

The plugin uses a WordPress REST route to send tracking-related information from the front end to the server:

- **Endpoint**  
  `/wp-json/sdas/v1/track`

- **How it works (high level)**  
  - The plugin's JavaScript notices a relevant action, such as a page view or form submit.
  - It sends a small, non-personal payload to `/wp-json/sdas/v1/track`.
  - WordPress receives the request and passes it into Tracking Hub's queue.
  - Tracking Hub then delivers normalized events to connected platforms.

As a site admin:

- You **do not normally call this endpoint yourself**. The plugin and related SDAS extensions handle it.
- You should avoid installing other plugins that interfere with core REST routing or aggressively block `/wp-json` calls from your own pages.
- If your theme or another plugin implements custom lead flows, your developer can hook into this endpoint safely (see the Advanced section).

## Click IDs, attribution, and stable URLs

To understand which ads lead to real leads and form submissions, Tracking Hub uses click IDs and related parameters from landing page URLs.

### Which parameters matter

On landing pages, you often see marketing parameters like:

- **Google Ads click IDs**  
  - `gclid` for standard Google Ads clicks  
  - `gbraid` and `wbraid` for some privacy-preserving flows

- **Meta click IDs**  
  - `fbclid` from Meta (Facebook/Instagram) clicks

When a visitor first arrives, SDAS Tracking Hub reads these values and stores them so it can later connect a lead or conversion event back to the original click.

### Why stable URLs matter

For attribution to work reliably:

- **Do not strip click IDs too early**  
  Avoid rewriting URLs or stripping `gclid`, `gbraid`, `wbraid`, or `fbclid` before Tracking Hub has a chance to see them on the first page load.

- **Keep URL structures predictable**  
  Use stable URL patterns for:
  - Inventory list pages.
  - Vehicle detail pages.
  - Lead and contact forms.

- **Coordinate with front-end routing**  
  If your theme or plugins use single-page app style navigation or heavy JavaScript routing:
  - Ensure the original landing URL (with click IDs) is visible long enough for Tracking Hub to capture it.
  - Avoid navigation patterns that lose or hide those parameters during the initial page view.

<Callout kind="info">

If your privacy policy or ad platform rules require you to remove click IDs after a certain time, confirm that your SDAS settings and consent behavior match that policy. Do not expose click IDs to unrelated third-party scripts, and do not keep them longer than allowed.

</Callout>

## How Tracking Hub connects to inventory and lead flows

Tracking Hub becomes most useful when it connects real visitor journeys across inventory browsing and lead submission on your site.

### Inventory pages

Coordinate with the [Inventory (front end)](/plugins/sdas-inventory-front-end) plugin so that:

- Inventory list pages use consistent templates and URLs.
- High-intent views (such as filtered lists) are easy to recognize as specific page types.
- You avoid creating multiple, slightly different versions of the same list view unless you have a clear tracking reason.

### Vehicle detail pages (VDP)

Vehicle detail pages typically represent strong shopping intent. To help Tracking Hub:

- Use consistent templates for VDPs so the system can recognize them.
- Keep vehicle identifiers and URL patterns stable.
- Coordinate with [Vehicle Filters (front end)](/plugins/sdas-vehicle-filters-front-end) so filter behavior and page state remain predictable.

### Lead and contact forms

Lead submissions are often your primary conversions. Together with [Forms Extension (front end)](/plugins/sdas-forms-extension-front-end):

- Use a small number of standardized lead and contact forms where possible.
- Avoid multiple, near-duplicate forms that behave differently from a tracking perspective.
- Confirm that:
  - Successful submissions always redirect to a clear confirmation or thank-you state.
  - The form behaves the same whether tracking is on or off.

Tracking Hub attaches to the submission event and passes anonymized metadata server-side while PII stays within your secure WordPress and CRM flows.

## Related Vehicle Tracking (Carousels and Widgets)

If your site includes "Related Vehicles", "Similar Vehicles", or "Recently Viewed" carousels outside the main inventory grid, Tracking Hub can automatically track interactions with those elements and push rich vehicle data to the GTM `dataLayer`.

When a visitor interacts with a related vehicle element, the plugin:
1. Reads the stock number from the element (from its text, a data attribute, or a custom attribute).
2. Looks up the vehicle's full details (make, model, year, price) from a cached inventory file — this is fast and does not add server load.
3. If the vehicle is not in the cache (e.g., it was just added), it falls back to a live lookup.
4. Pushes a `relatedto_vehicle` event to the `dataLayer` with the full vehicle context attached.

This feature requires a developer to configure the CSS selectors and page paths where it should run. Once configured, it works automatically for every visitor.

## Practical admin test plan

Use this plan in a staging or test environment where SDAS Tracking Hub is enabled and connected to non-production properties. Work with whoever monitors SDAS admin screens or logs to confirm results.

<Steps>

<Step title="Check consent behavior" icon="shield">

- Open your staging site in a private or incognito window.
- When the consent banner appears, **deny** marketing or analytics tracking.
- Browse:
  - At least one inventory list page.
  - At least one vehicle detail page.
  - A lead or contact form page, and submit a test lead.
- Confirm:
  - Pages load quickly.
  - Forms submit successfully.
  - You do not see extra popups or errors about tracking.

Then update your consent to **allow** tracking (using the same banner or settings), and repeat the same journey.

Success signal: The user experience looks and feels the same with tracking off or on. The only difference is what Tracking Hub is allowed to send in the background.

</Step>

<Step title="Visit key inventory and VDP pages" icon="monitor">

- With tracking allowed, start from a "campaign-style" URL if your team has one (for example, a staging link that includes `gclid` or `fbclid` for testing).
- Visit:
  - A primary inventory list page (with typical filters applied).
  - At least one vehicle detail page from that list.
- Watch the address bar:
  - Confirm that click ID parameters (such as `gclid` or `fbclid`) appear on the landing page.
  - Confirm that later pages have sensible, stable URLs, even if the click IDs no longer appear.

Ask your developer or analytics owner to check the SDAS Tracking Hub admin or logs.

Success signal: They see server-side events for inventory and VDP views that match your test journey and, where expected, retain the original click/source information.

</Step>

<Step title="Submit a test lead" icon="file-text">

- With tracking still allowed, go to your main lead or contact form.
- Complete the form with clearly fake but realistic details (for example, "Test User", a test email at your domain, and a note like "SDAS tracking test").
- Confirm on the front end:
  - The form validates and submits without delay.
  - You see a thank-you message or confirmation page.

Then ask the person who monitors SDAS or your CRM to confirm:

- The lead reached the CRM or email destination as expected.
- Tracking Hub shows a conversion event or lead-related event for this submission.

Success signal: The lead appears in your CRM or email inbox, and Tracking Hub records exactly one conversion event for that submission in the admin or logs.

</Step>

<Step title="Repeat with consent off (optional)" icon="x-circle">

- Reset your browser session (new private window).
- Deny tracking in the consent banner.
- Repeat the same:
  - Inventory visit.
  - Vehicle detail page view.
  - Lead form submission.

Ask your monitoring contact to check again.

Success signal: The lead still arrives in your CRM or email, but Tracking Hub either does not record marketing events for this journey or records only what your policy allows for non-consenting visitors.

</Step>

</Steps>

## Advanced (developers)

The details in this section are primarily for developers integrating SDAS Tracking Hub with custom themes or plugins. Site admins can review them, but you do not need to change code to benefit from Tracking Hub.

### What Tracking Hub sends to external platforms

Tracking Hub sends normalized events from your WordPress site to external platforms server-side. Developers do not need to embed platform credentials or scripts in the front end.

<Callout kind="info">

Measurement IDs, customer IDs, API secrets, access tokens, and developer tokens live in SDAS admin settings and server-side secure storage. Front-end code never embeds or exposes these values.

</Callout>

#### GA4 via Measurement Protocol

For Google Analytics 4, Tracking Hub uses the Measurement Protocol endpoint:

- **Endpoint**  
  `https://www.google-analytics.com/mp/collect?measurement_id=...&api_secret=...`

- **Behavior**  
  When an eligible event is queued, the server posts a payload containing:
  - A client identifier derived from cookies or attribution context.
  - Event name and parameters (for example, page type, lead status, inventory identifiers).
  - Timestamps and traffic source data when available.

#### Google Ads via offline uploadClickConversions

For Google Ads, Tracking Hub sends conversions as offline events using the Google Ads API `uploadClickConversions` call:

- **Endpoint pattern**  
  `https://googleads.googleapis.com/v22/customers/{customer_id}:uploadClickConversions`

- **Behavior**  
  When the server has a valid Google Ads click identifier (`gclid`, `gbraid`, or `wbraid`) and a qualifying event:
  - It queues a conversion payload tied to that click ID.
  - It includes conversion times and additional allowed context.
  - It authenticates using OAuth and a developer token configured in admin settings.

#### Meta via Conversions API (CAPI)

For Meta (Facebook), Tracking Hub sends events via the Conversions API:

- **Endpoint pattern**  
  `https://graph.facebook.com/v22.0/{pixel_id}/events`

- **Behavior**  
  When Meta-related context is available (`fbclid`, or cookies equivalent to `fbc` and `_fbp`), Tracking Hub:
  - Sends a server-side event with hashed user data where allowed.
  - Includes click identifiers and browser context for better attribution.
  - Authenticates using an access token configured in admin settings.

### REST tracking endpoint for custom front-end features

Front-end JavaScript in the plugin (and your custom code) can dispatch tracking-relevant events to a WordPress REST endpoint:

- **Endpoint**  
  `/wp-json/sdas/v1/track`

- **Behavior**  
  When a page view, form submission, or other interaction occurs:
  - The front end can call this endpoint with a minimal payload (event type, context, and identifiers).
  - The endpoint validates the request and hands it to the server-side queue.
  - Server-side workers then fan the event out to GA4, Google Ads, and Meta.

You usually do not call this endpoint directly; the plugin's scripts (`assets/js/tracking.js` and `form-tracking.js`) handle it. If you build custom front-end features, coordinate with the back-end team before sending custom events here.

### Consent gating with sdas_has_tracking_consent

Every tracking call from the front end must respect the visitor's consent status.

- **Server-side consent helper**  
  The plugin exposes a helper `sdas_has_tracking_consent()` that the back end can use when deciding whether to enqueue tracking or render tracking-related scripts.

- **Front-end expectations**  
  Coordinate so that:
  - Any code path that leads to `/wp-json/sdas/v1/track` only runs when consent is granted.
  - Consent changes (from banner or settings UI) update the underlying state used by `sdas_has_tracking_consent()`.

<Callout kind="info">

Treat consent as a single source of truth. Avoid duplicating consent flags in JavaScript when the back end already exposes a reliable consent state. When in doubt, assume tracking is not allowed until the consent state says otherwise.

</Callout>

### Code examples for developers

These examples show how to gate custom tracking by consent and post a minimal custom event to the Tracking Hub REST endpoint.

<Callout kind="info">

Only send non-PII event metadata from the browser, such as page type, inventory identifiers, or anonymized flags. Never include names, emails, phone numbers, or other personally identifiable information in custom payloads, and always respect the consent state before triggering tracking.

</Callout>

#### Gate custom front-end tracking by consent (PHP template)

Use the consent helper in your theme or plugin templates to only render custom tracking hooks when consent is granted.

<CodeGroup tabs="PHP template">

```php
<?php
// Example: Only output custom tracking markup when tracking consent is granted.
if ( function_exists( 'sdas_has_tracking_consent' ) && sdas_has_tracking_consent() ) :
    ?>
    <script>
      // Safe to run custom tracking-related JS here.
      // Avoid PII and rely on the REST endpoint for server-side tracking.
      window.sdasCustomTrackingEnabled = true;
    </script>
    <?php
endif;
?>
```

</CodeGroup>

#### Post a minimal custom event to the Tracking Hub REST endpoint (JavaScript)

From your custom JavaScript, send a small, non-PII event payload to `/wp-json/sdas/v1/track` after your own logic confirms that tracking is enabled.

<CodeGroup tabs="JavaScript">

```javascript
// Example: post a minimal, non-PII event to the SDAS Tracking Hub REST endpoint.
async function sendSdasCustomEvent() {
  // Optionally check a flag that your PHP template set when consent is granted.
  if (!window.sdasCustomTrackingEnabled) {
    return
  }

  const payload = {
    event_type: "custom_inventory_view",
    context: {
      page_type: "inventory_list",
      inventory_source: "primary",
      // Include only non-PII metadata that helps attribution.
      experiment_variant: "variant_b"
    }
  }

  try {
    const response = await fetch("/wp-json/sdas/v1/track", {
      method: "POST",
      headers: {
        "Content-Type": "application/json"
      },
      credentials: "same-origin",
      body: JSON.stringify(payload)
    })

    if (!response.ok) {
      // Optionally log for debugging in non-production environments.
      console.warn("SDAS tracking request failed with status", response.status)
    }
  } catch (error) {
    // Do not block the UX; treat tracking failures as non-fatal.
    console.warn("SDAS tracking request error", error)
  }
}

// Call this from an interaction you have defined as a trigger point.
document.addEventListener("DOMContentLoaded", () => {
  const inventoryList = document.querySelector("[data-sdas-inventory-list]")
  if (inventoryList) {
    sendSdasCustomEvent()
  }
})
```

</CodeGroup>

#### Example staging URL with click IDs

When you test attribution in staging, use URLs that contain realistic click identifiers and keep them stable across navigation.

<CodeGroup tabs="Test URL">

```text
https://staging.dealer-site.test/inventory/suvs?
  gclid=TEST_GCLID_123&
  gbraid=TEST_GBRAID_456&
  fbclid=TEST_FBCLID_789
```

</CodeGroup>

Stable URLs ensure that SDAS Tracking Hub can reliably capture `gclid`, `gbraid`, and `fbclid` on the initial landing page and associate later server-side events with the correct marketing touchpoint, even when the user navigates through additional pages.
