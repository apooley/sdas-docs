## How SDAS Tracking Hub affects the front end

SDAS Tracking Hub sends server-side events to GA4, Google Ads, and Meta, handling attribution, PII normalization, and queued delivery with retries in the background.

From the front end, this tracking is mostly invisible. You do not manage tags or pixels directly, but you do control when important events should happen, how consent is collected, and whether pages expose enough context for accurate attribution.

<Callout kind="info">

SDAS Tracking Hub centralizes tracking for analytics and ads platforms, but privacy still starts at the browser. Front-end implementations must respect consent, avoid exposing sensitive data in the DOM or URLs, and align with your site's legal requirements.

</Callout>

## What it actually sends (platform integrations)

SDAS Tracking Hub sends a single normalized event from your site and fans it out server-side to GA4, Google Ads, and Meta, using each platform's API.

<Callout kind="info">

Measurement IDs, customer IDs, API secrets, access tokens, and developer tokens live in SDAS admin settings and server-side secure storage. Front-end code never embeds or exposes these credentials.

</Callout>

### GA4 via Measurement Protocol

For Google Analytics 4, SDAS Tracking Hub uses the Measurement Protocol endpoint:

- **Endpoint**  
  `https://www.google-analytics.com/mp/collect?measurement_id=...&api_secret=...`

- **Behavior**  
  When an eligible event is queued, the server posts a payload containing:
  - A client identifier derived from cookies or attribution context
  - Event name and parameters (for example, page type, lead status, inventory identifiers)
  - Timestamps and traffic source data when available

- **What front-end teams should know**  
  You do not load the GA4 JavaScript library or gtag on the page. Your job is to ensure that:
  - Event trigger points are well defined and stable.
  - Client identifiers and attribution context (such as landing URLs) remain consistent across the journey.

### Google Ads via offline uploadClickConversions

For Google Ads, SDAS Tracking Hub sends conversions as offline events using the Google Ads API `uploadClickConversions` call:

- **Endpoint pattern**  
  `https://googleads.googleapis.com/v22/customers/{customer_id}:uploadClickConversions`

- **Behavior**  
  When the server has a valid Google Ads click identifier (for example, `gclid`, `gbraid`, or `wbraid`) and a qualifying event:
  - It queues a conversion payload tied to that click ID.
  - It includes conversion times and additional context that the platform allows.
  - It authenticates using OAuth and a developer token configured in admin settings.

- **What front-end teams should know**  
  Front-end code never talks to the Google Ads API directly. Instead, you help by:
  - Preserving URL parameters like `gclid`, `gbraid`, and `wbraid` when users navigate.
  - Avoiding SPA flows that drop or rewrite these parameters without a coordinated strategy.

### Meta via Conversions API (CAPI)

For Meta (Facebook), SDAS Tracking Hub sends events via the Conversions API:

- **Endpoint pattern**  
  `https://graph.facebook.com/v22.0/{pixel_id}/events`

- **Behavior**  
  When Meta-related context is available (such as `fbclid`, or cookies equivalent to `fbc` and `_fbp`), SDAS Tracking Hub:
  - Sends a server-side event with hashed user data where allowed.
  - Includes click identifiers and browser context for better attribution.
  - Authenticates using an access token configured in admin settings.

- **What front-end teams should know**  
  You typically do not load the Meta pixel script when SDAS Tracking Hub is in charge. However, you should:
  - Ensure the consent state clearly covers Meta and other marketing pixels.
  - Avoid clearing or rewriting cookies and URL parameters that Meta uses unless required by consent or legal policy.

## What front-end teams influence

Front-end developers and site builders influence three main areas that determine whether SDAS Tracking Hub works correctly and compliantly:

- **Event trigger points**: When key user actions (page views, form submissions, navigation flows) should count as trackable events.
- **Consent and privacy behavior**: How and when visitors grant or deny tracking, and how that decision affects downstream tracking.
- **Attribution context**: What contextual information pages expose so server-side tracking can attribute events to the right visit and marketing source.

<Tabs>

<Tab title="What visitors see" icon="monitor">

Visitors do not see SDAS Tracking Hub directly. They experience:

- **Fast pages without extra tracking scripts**: Because SDAS Tracking Hub runs on the server, you avoid loading multiple third-party pixels for GA4, Google Ads, and Meta on every page.
- **A consistent consent experience**: The consent banner, settings dialog, or cookie preferences are the only visible tracking-related UI.
- **Normal forms and navigation**: Lead forms, search flows, and other UI behave as expected; tracking attaches to events that already happen as part of the UX.

Your goal on the front end is to keep the UX clean and responsive while ensuring that key points in the journey are clearly defined so tracking can attach to them.

</Tab>

<Tab title="What front-end teams control" icon="code">

Front-end teams coordinate with back-end and analytics owners to define:

- **Which interactions are trackable**  
  Decide which form submissions, page views, or navigation events are meaningful enough to trigger server-side tracking. You do not name the events here; you agree on *where* in the UI they should fire.

- **When tracking is allowed**  
  Ensure that any client-side trigger checks the visitor's consent status before it signals that an event is eligible for tracking. If the visitor has not consented, avoid calling any hooks that lead to server-side tracking.

- **What context is available**  
  Make sure pages and forms expose enough structured context that the back end can use for attribution and PII-safe tracking, such as:
  - Clear page types (for example, detail vs list vs checkout)
  - Stable identifiers rendered in the page markup when appropriate
  - Clean URLs that do not leak sensitive information

- **Error-tolerant UI behavior**  
  Design your UI so that if tracking fails server-side (queue, retry, or cleanup behavior), visitors still see successful form submissions or navigation. Tracking should never block or slow down the UX.

</Tab>

<Tab title="Testing" icon="check-circle">

Testing SDAS Tracking Hub from the front end focuses on **behavior**, not on direct network calls to GA4, Google Ads, or Meta.

You validate that:

- **Event trigger points line up with the UX**  
  When a user completes a journey step you care about, the corresponding server-side event is queued — without double-firing on refresh or back/forward navigation.

- **Consent is honored**  
  When tracking is off, journeys still work but do not produce trackable events. When tracking is on, journeys behave the same but do produce events.

- **Attribution context is stable**  
  URLs and page state remain consistent across the journey so server-side attribution can connect early touchpoints with later events without relying on fragile front-end hacks.

The next section walks through a safe test plan you can run in any non-production environment.

</Tab>

</Tabs>

## Front-end touchpoints

While the heavy lifting happens server-side, SDAS Tracking Hub exposes a small number of front-end touchpoints you should understand.

### REST tracking endpoint

Front-end JavaScript in the plugin can dispatch tracking-relevant events to a WordPress REST endpoint:

- **Endpoint**  
  `/wp-json/sdas/v1/track`

- **Behavior**  
  When a page view, form submission, or other interaction occurs:
  - The front end can call this endpoint with a minimal payload (event type, context, and identifiers).
  - The endpoint validates the request and hands it to the server-side queue.
  - Server-side workers then fan the event out to GA4, Google Ads, and Meta.

You usually do not call this endpoint directly; the plugin's scripts (`assets/js/tracking.js` and `form-tracking.js`) handle it. However, if you build custom front-end features, coordinate with the back-end team before sending custom events here.

### Consent gating with sdas_has_tracking_consent

Every tracking call from the front end must respect the visitor's consent status.

- **Server-side consent helper**  
  The plugin exposes a helper `sdas_has_tracking_consent()` that the back end can use when deciding whether to enqueue tracking or render tracking-related scripts.

- **Front-end expectations**  
  Coordinate with your back-end colleagues so that:
  - Any code path that leads to `/wp-json/sdas/v1/track` only runs when consent is granted.
  - Consent changes (from banner or settings UI) update the underlying state used by `sdas_has_tracking_consent()` without requiring fragile per-component flags.

<Callout kind="info">

Treat consent as a single source of truth. Avoid duplicating consent flags in JavaScript when the back end already exposes a reliable consent state. When in doubt, assume tracking is not allowed until the consent state says otherwise.

</Callout>

### Click ID capture and stable URLs

SDAS Tracking Hub stores marketing click identifiers and attribution context for later server-side sends. This includes:

- **Google Ads click IDs**  
  - `gclid` for standard Google Ads clicks  
  - `gbraid` and `wbraid` for certain privacy-preserving flows

- **Meta click IDs and browser identifiers**  
  - `fbclid` from landing page URLs  
  - Cookie-like values equivalent to `fbc` and `_fbp` when available

When a user first lands on the site, the plugin reads these identifiers from the URL and browser context and stores them for later use in server-side events.

Stable URLs are critical for this to work well:

- Avoid stripping `gclid`, `gbraid`, `wbraid`, or `fbclid` from the URL before SDAS Tracking Hub can read them.
- If you rely on client-side routing, coordinate so that:
  - The initial landing URL with click IDs is visible to the server or tracking scripts.
  - Subsequent navigation does not unintentionally lose attribution parameters needed for offline conversion uploads.

<Callout kind="info">

If legal or privacy requirements force you to remove click IDs or other identifiers, confirm that your SDAS settings and consent flows match that policy. Never keep identifiers longer than your policy allows, and avoid exposing them to third-party scripts unnecessarily.

</Callout>

## Coordinate event trigger points

Event trigger points are the **moments in the UI** that should result in server-side events being queued for GA4, Google Ads, and Meta.

Work with your analytics or marketing owner to document:

- **Which pages matter**: For example, landing pages, key search flows, or high-intent detail pages.
- **Which interactions matter**: For example, form submissions or specific navigation actions.
- **Which edge cases to ignore**: For example, repeated submissions, previews, or internal QA journeys that you do not want to track.

Once you agree on these points, implement stable, testable hooks in your front-end code or templates that the back end can listen for, without leaking PII in the process.

## Consent and privacy expectations

Consent management affects whether the server is allowed to send events to GA4, Google Ads, and Meta at all.

On the front end:

- **Do not hard-code assumptions about tracking**  
  Always treat tracking as conditional. Build your forms and pages so they work identically whether tracking is enabled or disabled for a specific visitor.

- **Centralize consent checks**  
  Avoid scattering one-off flags across templates. Use a single path to read the visitor's consent status and base any tracking-related decisions on that.

- **Avoid PII in URLs or client-side identifiers**  
  Let SDAS Tracking Hub handle PII normalization server-side. On the front end, avoid encoding names, emails, phone numbers, or similar sensitive data in URLs, query strings, or client-side identifiers.

- **Keep the banner and settings usable**  
  Make it obvious how visitors can change their consent, and ensure updates propagate to your tracking logic without page reloads that could cause double-counting.

## Provide attribution-friendly context

Attribution connects a final event (for example, a lead submission) to earlier interactions and traffic sources.

Front-end decisions that help attribution include:

- **Stable URLs**  
  Keep URL structures consistent between environments and across releases so attribution logic does not break when routes change. Avoid encoding sensitive details directly into the URL.

- **Clear page semantics**  
  Use consistent templates and markup for the same types of pages. When the back end parses the request, it should be able to tell what kind of page the user saw without relying on fragile CSS selectors.

- **Predictable navigation patterns**  
  Avoid JavaScript flows that change significant content without updating the URL unless you coordinate with the back end on how those states are represented. Where possible, reflect important navigation state in a way the server can observe.

- **Resilient forms**  
  Design forms so that they validate and submit predictably, even if the tracking queue is temporarily delayed or unavailable. The server-side queue, retries, and cleanup should not affect whether users see a confirmation.

## How it relates to inventory and lead flows

SDAS Tracking Hub becomes most valuable when it connects real visitor journeys across inventory browsing and lead submission flows.

Common front-end trigger locations include:

- **Inventory lists**  
  High-intent impressions of vehicle list pages, especially when filtered, often count as meaningful page views. Coordinate with the [SDAS Inventory](/plugins/sdas-inventory-front-end) front-end to expose consistent list templates and URLs.

- **Vehicle detail pages (VDP)**  
  Visits to individual vehicle detail pages usually represent strong intent and are prime candidates for GA4 and ads events. Align with the [Vehicle Filters (front end)](/plugins/sdas-vehicle-filters-front-end) behavior so that filter state and vehicle identifiers remain clear.

- **Lead and contact forms**  
  Successful submissions from forms on inventory or generic lead pages are key conversion points. Work with the [Forms Extension (front end)](/plugins/sdas-forms-extension-front-end) to ensure that:
  - The submission events are stable and debounced.
  - PII is handled only through server-side processing while tracking attaches to the event metadata.

When you design or change these flows, check how your changes affect event trigger points defined in SDAS Tracking Hub settings (for example, under generic "Events" or plugin-specific options in SDAS settings).

## Safe front-end test plan

Use this test plan in a staging or test environment where SDAS Tracking Hub is enabled and connected to non-production GA4, Google Ads, and Meta properties.

<Steps>

<Step title="Confirm consent and baseline behavior" icon="shield">

- Open the site in a private browsing window with a clean profile.
- Interact with the consent banner or settings to **deny** tracking.
- Navigate through key pages and submit at least one test form.
- Confirm that:
  - The UI behaves normally.
  - No additional popups or delays appear that depend on tracking.
- Then update your consent to **allow** tracking and repeat the same journey.

Success signal: The UX is identical with or without tracking consent; any differences are limited to what is allowed to be tracked server-side.

</Step>

<Step title="Verify event trigger points" icon="terminal">

- With tracking allowed, walk through each defined event trigger point:
  - Important page views
  - High-value interactions such as key form submissions
- For each trigger point:
  - Perform the action once.
  - Refresh the page and confirm that the action does not trigger again unless the user repeats it.
- Coordinate with whoever monitors the SDAS Tracking Hub server-side logs or admin views to confirm that:
  - Exactly one event is queued per user action.
  - Events appear with the expected timestamps and context.

Success signal: Every agreed trigger point results in one server-side event per real user action, without duplicates caused by refreshes or navigation quirks.

</Step>

<Step title="Validate attribution context stability" icon="bar-chart">

- Start from an entry page that simulates a real acquisition source (for example, by opening a link that represents a campaign or referrer your team uses for testing).
- Move through your normal user journey toward a key event, such as completing a form.
- Note:
  - How the URL changes at each step.
  - Whether the page type or state is clear from the rendered markup.
- After completing the journey, work with the back-end or analytics owner to check that:
  - The final event is associated with the expected earlier page views.
  - The journey is distinguishable from other flows you do not want to treat the same way.

Success signal: Server-side tracking shows a coherent path from entry to event, without missing or ambiguous steps caused by front-end routing or rendering.

</Step>

</Steps>
