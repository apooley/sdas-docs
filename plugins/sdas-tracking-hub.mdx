## Overview

SDAS Tracking Hub provides a centralized system for managing and dispatching tracking events from your WordPress site to various analytics and advertising platforms. It captures front-end events, processes them server-side, and forwards them to services like Google Analytics 4 (GA4), Google Ads, and Meta (Facebook) with robust queuing and retry mechanisms. This ensures reliable data delivery for accurate reporting and campaign optimization.

## Admin Navigation and Interface

All SDAS Tracking Hub functionalities are accessible through the centralized **SDAS** menu in the WordPress admin sidebar. Navigate to **SDAS -> Tracking Hub** to access the main configuration and monitoring interface.

The Tracking Hub interface is organized into four distinct tabs:

-   **Overview**: Provides a high-level summary of the Tracking Hub's status, including any critical alerts or warnings regarding its operation.
-   **Settings**: Configure the core behavior of the Tracking Hub, including API credentials for external platforms (Google Ads, Meta, GA4).
-   **Event Config**: Define and manage the specific tracking events that the hub will process.
-   **Logs**: Displays a detailed log of all tracking events processed by the hub, including their delivery status.

## Configuring Tracking Events

The **Event Config** tab is where you map internal SDAS events (like a form submission or a VDP view) to the specific formats required by external platforms.

### Workflow: Creating a New Event

1.  **Add New Event**: Click the "Add New Event" button in the Event Config tab.
2.  **Select Trigger**: Choose the internal SDAS event that should trigger this tracking payload. Common triggers include:
    -   `sdas_lead_submitted` (Fires when a WPForms lead is captured)
    -   `sdas_vdp_viewed` (Fires when a visitor views a vehicle detail page)
    -   `sdas_search_performed` (Fires when a visitor uses the inventory search)
3.  **Select Destination**: Choose where this event should be sent (e.g., GA4, Google Ads, Meta Conversions API).
4.  **Map Fields**: Depending on the destination, you will need to map internal data to the platform's required fields.
    -   *Example for GA4 `generate_lead`*: Map the internal `lead_value` to the GA4 `value` field, and set the `currency` to your local currency (e.g., `CAD`).
    -   *Example for Meta `Lead`*: Ensure the `email` and `phone` fields are mapped so Meta can perform advanced matching. The Tracking Hub automatically hashes these values before sending them to Meta.
5.  **Set Conditions (Optional)**: You can restrict when an event fires. For example, you might only want to send a Google Ads conversion if the lead came from a specific form ID.
6.  **Save and Enable**: Save the configuration and ensure the toggle switch is set to "Active".

### Example: A Correctly Configured GA4 Lead Event

A typical GA4 `generate_lead` event configuration looks like this:
-   **Trigger**: `sdas_lead_submitted`
-   **Destination**: Google Analytics 4
-   **Event Name**: `generate_lead`
-   **Parameters**:
    -   `value`: `[lead_estimated_value]` (Dynamic tag)
    -   `currency`: `CAD`
    -   `form_id`: `[form_id]` (Dynamic tag)

## Verifying Event Delivery (Logs)

The **Logs** tab is your primary tool for QA and troubleshooting. Because the Tracking Hub processes events server-side via Action Scheduler, you cannot rely solely on the browser's network tab to verify delivery.

### Workflow: Testing a Lead Event

1.  **Submit a Test Lead**: Go to your public website and submit a test lead through one of your configured forms.
2.  **Check the Logs Tab**: Navigate to **SDAS -> Tracking Hub -> Logs**.
3.  **Find the Event**: Look for a new entry corresponding to your test submission. The log table shows the Event Name, Destination, Status, and Timestamp.
4.  **Inspect the Payload**: Click "View Details" on the log entry. This opens a modal showing the exact JSON payload that was sent to the external platform, as well as the HTTP response code received back.
5.  **Verify Success**: A status of `Success` (HTTP 200/204) means the platform accepted the event.
6.  **Troubleshoot Failures**: If the status is `Failed`, the details modal will show the error message returned by the platform (e.g., "Invalid API Key" or "Missing Required Parameter"). The Tracking Hub will automatically attempt to retry failed events based on its internal schedule.

## Server-Side Tracking Endpoint

The Tracking Hub exposes a dedicated REST API endpoint for receiving tracking events from your front-end. 

-   **Method**: `POST`
-   **Route**: `/wp-json/sdas/v1/track`
-   **Purpose**: To receive JSON payloads containing tracking event data from your website's front-end (e.g., VDP views, button clicks).

Front-end implementations (like the SDAS Tracking Hub front-end plugin) send event data to this endpoint. The hub then processes this data, normalizes it, and dispatches it to the configured external platforms according to your Event Config rules.

### Related Vehicle Tracking

The Tracking Hub includes a powerful feature for tracking interactions with vehicles outside of the standard inventory grid or VDP (e.g., a "Related Vehicles" carousel or a "Recently Viewed" widget).

It can automatically extract a stock number from a DOM element, resolve the full vehicle details (make, model, year, price) via a fallback REST endpoint (`/wp-json/sdas/v1/vehicle-by-stock`), and push a rich `relatedto_vehicle` event to the `dataLayer`.

Developers can configure this via the `sdas_tracking_hub_related_vehicle_configs` PHP filter, specifying the CSS selector, the source of the stock number (text, attribute, or dataset), and the pages where the tracking should run.
