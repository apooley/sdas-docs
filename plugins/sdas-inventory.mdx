## Overview

sdas-inventory manages the core vehicle inventory in WordPress and exposes reusable building blocks for search, display, and enrichment of vehicle data.

The plugin defines and owns the vehicle custom post type, provides schema and description-generation services, and ships frontend templates for common inventory experiences such as grids, search, and comparison.

<Callout kind="info">

sdas-inventory depends on SDAS Core for shared constants (such as `SDAS_VEHICLE_POST_TYPE`) and common services. Load SDAS Core before activating or using sdas-inventory.

</Callout>

## Capabilities at a glance

Use sdas-inventory when you need:

- A canonical **vehicle custom post type** (CPT) and schema.
- Frontend **templates/shortcodes** for:
  - Vehicle grid and search
  - Gallery and slider views
  - Vehicle comparison
  - Payment calculator
  - Quick view overlays
  - Similar vehicles and recently viewed vehicles
  - Save vehicle and related user interactions
- **Awards enrichment** using external sources (AJAC, IIHS, NHTSA) scheduled via Action Scheduler.
- A **vehicle description generator** that produces human-readable copy from structured data.

#### Inventory grid and search

Use the inventory grid and search templates to list vehicles with filters and sorting.

Typical use cases:

- Primary **inventory listing page** (grid or list layout).
- **Search results** page combining query, filters, and pagination.
- Embedded **inventory sections** on marketing pages.

The templates generally:

- Query the vehicle CPT defined by SDAS Core and sdas-inventory.
- Apply common filters such as make, model, year, price, body style, and mileage.
- Render configurable card layouts and metadata badges.

#### Gallery and slider

Gallery and slider templates render media-focused views of one or more vehicles:

- **Single-vehicle gallery** showing images and possibly videos of a specific vehicle.
- **Slider / carousel** highlighting featured or promoted vehicles.

These templates are typically used on:

- Vehicle detail pages.
- Homepages and landing pages showcasing featured inventory.

#### Comparison

Comparison templates allow users to evaluate multiple vehicles side by side.

Core behavior:

- Accept a selection of vehicle IDs.
- Render a **column-per-vehicle** view with shared attributes (price, powertrain, dimensions, features) aligned in rows.
- Integrate with save or selection mechanisms exposed by the plugin.

#### Payment calculator

The payment calculator supports basic financing estimates for a given vehicle:

- Uses vehicle price and optional fees or taxes as inputs.
- Lets users adjust term length, interest rate, and down payment.
- Outputs estimated monthly or bi-weekly payments.

You can embed the calculator on:

- Vehicle detail pages next to pricing.
- Finance or credit application pages.

#### Quick view

Quick view templates provide an overlay or modal view of key vehicle details without a full page load.

Typical use:

- Attach quick view triggers to vehicle cards in grids or sliders.
- Render a lightweight summary: headline, price, key specs, and gallery preview.

Quick view templates generally rely on the same vehicle CPT and schema data as the full detail page, but with a reduced field set.

#### Similar and recently viewed vehicles

The plugin exposes templates for:

- **Similar vehicles**: suggest alternative vehicles based on shared attributes such as model, price band, or body style.
- **Recently viewed vehicles**: track and show vehicles a visitor has interacted with recently.

You can integrate these templates:

- At the bottom of vehicle detail pages.
- In sidebars or footers for cross-selling.

#### Save vehicle and stats/awards

Additional templates and hooks support:

- **Save vehicle**: allow authenticated or cookie-based users to mark vehicles as favorites for later review.
- **Stats**: surface aggregated information about a vehicle (for example, fuel economy, safety ratings) fed by schema and awards data.
- **Awards**: display badges and labels from AJAC, IIHS, NHTSA enrichment workflows.

### Vehicle custom post type

sdas-inventory acts as the core manager for the vehicle CPT that SDAS Core defines.

<Callout kind="info">

The actual CPT registration and the `SDAS_VEHICLE_POST_TYPE` constant live in SDAS Core. sdas-inventory builds schema, templates, and enrichment around that CPT.

</Callout>

Key points:

- Vehicles are stored as **posts** under the shared vehicle CPT.
- The plugin expects structured metadata for vehicle attributes such as year, make, model, trim, engine, drivetrain, mileage, and pricing.
- Frontend templates and generators read from this structured data rather than free-form content.

Because the CPT is shared across the SDAS ecosystem, other plugins can integrate with the same vehicle records without duplicating data.

### Vehicle schema

The plugin includes a dedicated schema helper:

- `class-sdas-vehicle-schema.php`

This class encapsulates:

- A **normalized schema** for vehicle attributes (for example, powertrain, dimensions, options).
- Accessors for reading and writing schema fields on the vehicle CPT.
- Potential support for structured markup (for example, JSON-LD) mapped from vehicle data.

Use this class when you:

- Create or update vehicles programmatically.
- Need a consistent way to read schema fields in custom templates.
- Integrate third-party systems with the SDAS vehicle data model.

### Vehicle description generator

Content generation is handled by:

- `class-vehicle-description-generator.php`

This class builds human-readable descriptions from the structured vehicle schema:

- Uses key attributes (year, make, model, trim) as the base.
- May incorporate highlights such as mileage, options, or awards.
- Outputs a description suitable for the post content area or dedicated excerpt fields.

Typical usage patterns:

- Generate descriptions automatically when creating or importing vehicles.
- Regenerate descriptions when important schema fields change.
- Provide a base description that editors can refine manually.

### Awards scraping and enrichment

sdas-inventory enriches vehicles with award and safety information from:

- **AJAC**
- **IIHS**
- **NHTSA**

Awards scraping typically:

- Pulls data from upstream sources on a schedule.
- Maps external identifiers (for example, make, model, trim, year) to your vehicle records.
- Saves awards or ratings into vehicle meta or schema fields for later display in templates.

### Action Scheduler integration

The plugin uses **Action Scheduler** to orchestrate awards workflows.

<Callout kind="info">

Ensure Action Scheduler is installed and working in your environment. Awards scraping and enrichment rely on queued actions rather than running inline on page loads.

</Callout>

Action Scheduler is used to:

- Queue **scrape jobs** for external awards sources.
- Schedule **update tasks** that refresh awards across your inventory.
- Handle **retry logic** for failed or time-limited requests.

Operational considerations:

- Monitor the Action Scheduler queue for long-running or failed awards jobs.
- Run WP-Cron or a real cron trigger to process scheduled actions consistently.
- Evaluate run frequency for awards updates so you balance freshness with external rate limits.

### Operational footprint

From an operations perspective, sdas-inventory:

- Adds scheduled tasks and background jobs for awards scraping and updates.
- Introduces additional database reads/writes for vehicle schema, awards, and description generation.
- Relies on SDAS Core for stable CPT definitions and potentially shared caching or API clients.

When deploying to production:

- Validate that background processing is functioning (no growing backlog of pending awards actions).
- Confirm that inventory pages and vehicle detail pages load within acceptable performance targets after enabling inventory features.
