## Overview

SDAS CSV Importer loads vehicle data from CSV files into the SDAS Inventory custom post type and runs the import work in the background. It also coordinates image re-processing and maintains its own database tables.

The main plugin file is `wp-content/plugins/sdas-csv-importer/sdas-csv-importer.php`. From there, the plugin wires the import pipeline, background jobs, and database layer.

<Callout kind="info">

SDAS CSV Importer depends on SDAS Inventory. Activate SDAS Inventory before you activate this plugin, otherwise vehicle import operations and related custom post types will not be available.

</Callout>

## Core responsibilities

SDAS CSV Importer covers four main areas:

- Parsing and importing vehicle CSV data into SDAS Inventory
- Running import work through background processing
- Coordinating image re-processing and ReImage API integration
- Managing scheduled jobs and plugin-specific database tables

Use this page as a reference when you need to trace import behavior, debug cron jobs, or locate the correct implementation file during development.

  - An **hourly job** dedicated to processing CSV imports
  - A **daily job** that cleans up plugin-related data

  The hourly job uses the cron hook `sdas_csv_import_hourly`. When this hook fires, the plugin advances the import queue using `ImportProcessor.php`, processing pending rows until the batch completes or limits are reached.

## Image re-processing and ReImage integration

Image work is handled by `ReImageProcessor.php`, which integrates with ReImage and respects existing optimization state such as WPCompress. The processor is responsible for re-processing images associated with imported vehicles.

When images require re-processing, the importer uses `ReImageProcessor.php` to perform the work in the background, coordinating with any relevant optimization or compression state that already exists in the site.

`sdas_csv_get_reimage_api_key()` retrieves the ReImage API key for use by the image processor and migrates legacy keys into SDAS Core secure token storage when necessary. This avoids storing keys directly in plain text configuration.

<Callout kind="info">

If you rotate your ReImage API key, update it through the mechanism that feeds `sdas_csv_get_reimage_api_key()`. The helper will continue to store and retrieve the key from SDAS Core secure storage, so code that calls the helper does not change.

</Callout>

## Database tables

SDAS CSV Importer maintains dedicated database tables to support its import queue and related metadata. The method `Database::create_tables()` encapsulates table creation and upgrades.

Typical call sites for `Database::create_tables()` include plugin activation or upgrade routines. When you debug schema issues or add new fields for import tracking, update this method and run the relevant activation or upgrade flow.

If you clone environments or restore from backups, confirm that the tables created by `Database::create_tables()` exist and match the current plugin version before running large CSV imports.

## Operational notes

<Callout kind="info">

SDAS CSV Importer assumes that WordPress pseudo-cron is running reliably. If your site uses real cron via a system scheduler, ensure it triggers `wp-cron.php` frequently enough for hourly and daily jobs to run on time.

</Callout>

For large CSV files or frequent imports:

- Monitor the runtime of the `sdas_csv_import_hourly` job to keep throughput in line with your expectations.
- Verify that the database tables managed by `Database::create_tables()` are present and not corrupted before pushing heavy workloads.
- Check `ImportProcessor.php`, `Cron/CronManager.php`, and `ReImageProcessor.php` first when debugging import behavior, timing, or image handling.

These touchpoints give you a consistent starting map when you extend the importer, add logging, or analyze production issues with CSV-based vehicle imports.
