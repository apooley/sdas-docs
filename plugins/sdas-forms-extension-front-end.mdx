## What SDAS Forms Extension does

SDAS Forms Extension turns your WPForms forms into dealership-ready vehicle leads without changing how visitors see or submit the form.

You still design and place forms using WPForms. The extension watches submissions in the background, recognizes which ones are vehicle leads, and turns them into structured leads with ADF output and an audit trail.

<Callout kind="info">

SDAS Forms Extension does **not** provide its own shortcodes or blocks. Embed forms using standard **WPForms** methods (shortcodes, blocks, widgets) and SDAS will interpret submissions behind the scenes.

</Callout>

<Callout kind="info">

**Prerequisites**

- A working WordPress site with **WPForms** installed and activated  
- **SDAS Core (MU-plugin)** installed and active in the `mu-plugins` directory  
- The **SDAS Forms Extension** plugin installed and activated  

If you are not sure whether SDAS Core is active, start with the [install and activate guide](/setup/installation).

</Callout>

## How it extends WPForms for vehicle leads

SDAS Forms Extension focuses on three things:

- Recognizing that a submission is a **vehicle lead**
- Checking that the lead is **complete enough** for sales follow-up
- Producing **structured outputs** (ADF + audit trail) for downstream systems

You keep using WPForms to:

- Design the form layout and labels
- Choose required fields
- Decide where the form appears on the site

The extension adds vehicle awareness and routing, so each configured form submission becomes a vehicle context, a contact record, and an event your CRM or SDAS plugins can use.

<Tabs>

<Tab title="What visitors see" icon="users">

Visitors interact with normal-looking WPForms forms:

- A **vehicle detail page form** like "Check availability", "Book a test drive", or "Ask about this vehicle"
- A **contact or sales form** with an optional "Vehicle you are interested in" field
- A **trade-in or finance form** with current vehicle details and contact info

From the visitor’s perspective, it is just a form:

- Familiar fields (name, email, phone, message, vehicle)
- Your usual styling from the theme
- A confirmation message or redirect after submit, as configured in WPForms

They never see SDAS internals, ADF payloads, or audit trails.

</Tab>

<Tab title="What admins configure" icon="settings">

As a WordPress admin, you:

- Decide **which WPForms forms count as vehicle lead forms**
- Add and arrange fields so sales has the details they need
- Use SDAS Forms Extension settings to **mark those forms as vehicle leads**
- Map which fields are **contact details**, **vehicle details**, and **lead meta**
- Place the forms on vehicle pages, landing pages, or contact pages

Once configured, every submission from those forms becomes a standardized SDAS lead without additional shortcodes or template changes.

</Tab>

</Tabs>

## Front-end touchpoints for admins

Forms Extension connects to WPForms and SDAS at a few key points. You do not need to write code for these, but it helps to know what they do.

### Submission handling (high level)

Whenever a visitor submits a WPForms form:

- WPForms validates and processes the submission
- SDAS Forms Extension checks whether this form is configured as a vehicle lead form
- If it is, the plugin:
  - Creates a **lead record** in SDAS
  - Builds an **ADF-style payload**
  - Records an **audit trail entry**

The submit button, confirmation message, and redirects still come from WPForms. From the visitor’s point of view, nothing changes.

### Address fields (Canada-focused behavior)

If you add an **Address** field in WPForms for lead forms, SDAS Forms Extension nudges the defaults toward **Canada-style** addresses.

As an admin, expect:

- Province-style labels instead of state-style labels
- Canadian postal code conventions
- Consistent labeling for dealership use in Canada

You still add Address fields through WPForms; the extension adjusts how they behave and display so they fit Canadian dealership expectations.

### Vehicle selector behavior

For sites that use **SDAS Inventory**, you can offer a **vehicle selector** inside WPForms forms so visitors can search for a specific vehicle as they type.

At an admin level, this means:

- You add a text field (often labeled "Vehicle you are interested in")
- Front-end code can call an SDAS-powered search to show matching vehicles in a dropdown
- Visitors can pick the exact vehicle, and SDAS ties the lead to that vehicle

You only need this when:

- You want visitors to search **across inventory** from a single form (for example, a general sales form)
- You want a **typeahead experience** rather than hard-coded dropdowns

Implementation details for this vehicle selector live in the **Advanced (developers)** section at the end of this page.

## Build a vehicle lead form (admin workflow)

Use this flow when you are a WordPress administrator setting up or adjusting lead forms.

<Steps>

<Step title="Create or choose a WPForms form" icon="file-text">

Start with the form that will collect vehicle leads:

- Create a **new form** in WPForms for vehicle enquiries, trade-ins, or finance requests, or  
- Open an **existing sales or contact form** you already use

The key is that this form is the one you expect visitors to use when they want to talk about a vehicle or buy/sell a car.

</Step>

<Step title="Add the right fields for a lead" icon="database">

In the WPForms builder, add fields so sales receives everything they need:

- **Contact details**:
  - Name (first and last)
  - Email address
  - Phone number
- **Vehicle details**:
  - Vehicle reference (stock number, VIN, or a vehicle selector field)
  - Or a free-text field like "Vehicle you are interested in" if nothing else is available
- **Lead context** (optional but helpful):
  - Type of request, such as "Check availability", "Book a test drive", or "Value my trade"
  - Notes or questions from the visitor

Mark any must-have details as **required** in WPForms so incomplete leads are less likely.

</Step>

<Step title="Place the form on key pages" icon="monitor">

Decide where visitors should see the form, then embed it using WPForms:

- On **vehicle detail pages** for high-intent "about this specific vehicle" leads
- On **inventory landing pages** to capture general interest
- On a **contact or sales page** for broader enquiries

Use whichever method fits your site:

- Add a **WPForms block** in the block editor  
- Paste the **shortcode** into page content  
- Use a **widget or template** placement provided by your theme

After publishing, visit the pages yourself to confirm:

- The form loads correctly
- The fields look right
- The form is easy to find from typical visitor paths (search, inventory, home page)

</Step>

<Step title="Mark the form as a vehicle lead form in SDAS" icon="settings">

In the WordPress admin, open the **SDAS Forms Extension** settings.

Find the area where you can manage forms and:

- Locate the WPForms form you just placed on your pages
- Mark or toggle it as a **vehicle lead form**

At a high level, this tells SDAS:

- Treat submissions from this form as **vehicle leads**, not generic contact forms
- Attach the lead to a vehicle context when possible
- Include these submissions in SDAS **lead reporting** and **audit trails**

You do not need to change the form’s shortcode or page placement; the change happens entirely in SDAS settings.

</Step>

<Step title="Map WPForms fields to SDAS lead fields" icon="settings">

Still within SDAS Forms Extension settings, map which form fields correspond to SDAS lead data.

Typical mappings include:

- **Contact fields**:
  - Name field → lead contact name
  - Email field → lead email
  - Phone field → lead phone
- **Vehicle fields**:
  - Stock number, VIN, or vehicle selector field → vehicle reference
  - Optional description or notes about the vehicle
- **Lead meta**:
  - Dropdown or radio buttons for "Inquiry type" → lead intent (test drive, availability, finance, trade-in)
  - Hidden fields or dropdowns for "Source" or "Campaign" → lead source metadata

Mapping tells SDAS which WPForms fields to read. You do **not** need to rename field IDs or add special field types; you only select the existing fields from your form.

</Step>

<Step title="Configure confirmations in WPForms" icon="monitor">

In the WPForms settings for your form, configure what visitors see after submitting:

- An **inline success message** (for example, "Thank you, a sales consultant will contact you shortly"), or
- A **redirect** to a thank-you or next-step page, such as a booking confirmation or FAQ page

You can also:

- Send **notification emails** to your sales or BDC team
- Customize the email subject line and content

These settings control the visitor-facing experience. SDAS will still process the lead in the background, regardless of which confirmation option you choose.

</Step>

<Step title="Test a submission as a visitor" icon="play-circle">

With the form configured and embedded:

- Open the page as a normal visitor (ideally in a private or incognito window)
- Fill out the form using realistic test data:
  - A sample name, email, and phone number
  - A real or realistic stock number or vehicle description
  - A short message or specific request
- Submit the form and confirm:
  - The page shows your chosen **success message** or redirect
  - Any notification emails arrive where expected

From the front-end perspective, this confirms that the form functions correctly.

</Step>

<Step title="Verify the lead in the admin" icon="check-circle">

Finally, verify that SDAS treated the submission as a vehicle lead:

- In WordPress admin, go to the SDAS **lead** or **reports** area (as configured for your install)
- Locate the test submission by:
  - Date and time
  - Contact name or email
  - Vehicle reference
- Confirm that:
  - The lead shows as a **vehicle lead**, not a generic contact
  - Key fields (contact details, vehicle, intent) are populated correctly
  - An **audit trail** entry exists showing how SDAS interpreted and processed the form

When this works, your front-end form is live as a vehicle-aware SDAS lead source.

</Step>

</Steps>

## Visitor experience vs admin controls

Seeing both sides helps you decide what to adjust when leads do not look the way you expect.

### What visitors experience

From a visitor’s perspective:

- The form looks like any other form on your site
- Labels, placeholders, and help text come from WPForms and your copy
- On submit, they:
  - Stay on the same page and see a success message, or  
  - Move to a thank-you or next-step page
- They are not exposed to ADF payloads, audit logs, or routing rules

If visitors are confused, you typically fix it by changing:

- Field labels and descriptions in WPForms
- Required fields and error messages
- Form placement and surrounding page content

### What you control as an admin

Behind the scenes, for forms marked as vehicle lead forms:

- SDAS recognizes each submission as a **vehicle lead**
- Field mappings convert WPForms fields into a **structured lead model**
- SDAS builds an **ADF-style payload** that other plugins (like CRM connectors) can send out
- A **lead audit trail** records:
  - Which form the visitor used
  - What data was captured
  - How SDAS interpreted and routed the lead

If leads are missing details or show the wrong intent, you usually fix it by:

- Updating which fields are required in WPForms
- Adjusting **field mappings** in SDAS Forms Extension settings
- Confirming that the correct form is marked as a **vehicle lead form**

## How it connects to other SDAS plugins

You do not configure these connections on every form, but knowing what they do helps with troubleshooting.

### Lead storage and events

When a submission qualifies as a vehicle lead, SDAS Forms Extension:

- Stores a structured lead in SDAS’s own lead table
- Triggers an internal **lead submitted event** that other SDAS plugins listen for

You do not see this event in the UI. Instead, you see its effects:

- Leads appear in SDAS reporting
- Connected plugins pick up the lead for further processing

### CRM Connector

If you use the **SDAS CRM Connector** plugin, it listens for new leads and:

- Queues them for delivery to your CRM
- Uses the ADF-style payload created by SDAS Forms Extension
- Sends leads asynchronously so visitors are not kept waiting

From an admin point of view:

- You configure CRM Connector separately
- Once active, it automatically reacts to leads from your SDAS-enabled forms

### Tracking Hub and attribution

With **SDAS Tracking Hub** active, SDAS can:

- Associate form submissions with earlier **sessions and campaigns**
- Use click IDs or UTM parameters captured during the visit
- Attribute each lead to a **source** or **channel**

This happens without extra code in the form. When configured, you can view:

- Which campaigns generate leads
- Which pages and paths are most effective

## Advanced (developers)

This section is optional for WordPress administrators. It explains the hooks and code-level integration points developers can use to customize behavior.

### Submission processing hook

SDAS Forms Extension listens to the WPForms submission lifecycle using the `wpforms_process_complete` action.

<CodeGroup tabs="PHP">

```php
<?php
// Inside SDAS Forms Extension (simplified example)
add_action(
    'wpforms_process_complete',
    function( $fields, $entry, $form_data, $entry_id ) {
        // Decide if this form is configured as a vehicle lead form,
        // then interpret and store the lead.
        sdas_forms_process_lead( $fields, $form_data, $entry_id );
    },
    10,
    4
);
```

</CodeGroup>

Developers can hook before or after this processing if they need to adjust how submissions are interpreted or logged.

### Vehicle selector AJAX endpoint

For sites with SDAS Inventory, developers can connect a front-end vehicle selector field to SDAS’s vehicle search using WordPress AJAX actions:

- Logged-in route: `wp_ajax_sdas_get_vehicles`
- Public route: `wp_ajax_nopriv_sdas_get_vehicles`

A typical JavaScript flow:

<CodeGroup tabs="JavaScript">

```javascript
// Example jQuery-based front-end search call
jQuery(document).on("input", ".sdas-vehicle-selector", function () {
  const query = jQuery(this).val().trim();

  if (query.length < 2) {
    return;
  }

  jQuery.ajax({
    url: window.ajaxurl, // or localized admin-ajax URL
    method: "POST",
    dataType: "json",
    data: {
      action: "sdas_get_vehicles",
      q: query
    },
    success(response) {
      // Render dropdown options using the returned vehicle list
      // for example: response.vehicles with stock, VIN, and title
      renderVehicleSuggestions(response);
    }
  });
});
```

</CodeGroup>

The `.sdas-vehicle-selector` input can live inside a WPForms HTML field or in a custom theme template.

### Lead event hook for integrations

After SDAS Forms Extension inserts a lead, it fires the `sdas_lead_submitted` action so other plugins or custom code can react.

<CodeGroup tabs="PHP">

```php
<?php
/**
 * After inserting a lead into wp_sdas_leads.
 */
do_action(
    'sdas_lead_submitted',
    $lead_id,   // Primary key from wp_sdas_leads
    $lead_data  // Structured lead array used for ADF and integrations
);
```

```php
<?php
// Inside SDAS CRM Connector (simplified)
add_action(
    'sdas_lead_submitted',
    function( $lead_id, $lead_data ) {
        // Queue lead for CRM sync using the structured lead payload.
        sdas_crm_queue_lead_sync( $lead_id, $lead_data );
    },
    10,
    2
);
```

</CodeGroup>

Use this hook to integrate with additional CRMs, marketing platforms, or internal systems while keeping the visitor experience unchanged.
